What "company brain" means in the current field

The winning shape: Git + markdown, not a vector DB alone

Multiple independent write-ups in the last few months converge on the same conclusion: plain markdown files in versioned folders beat a pure vector-search knowledge base for agent-facing company knowledge, because every major coding agent already reads Git natively and nothing else needs an API, auth, or SDK — "the file system IS the API." A pure embedding store is good for fuzzy recall over raw captures, but decisions and guidelines need to be an explicit, versioned, human-editable file an agent can cite by path.

The two-layer pattern some teams use: a vector layer over raw unprocessed captures (Slack threads, docs, competitor pages — the evidence), and a curated markdown wiki layer over promoted decisions (the guidelines and facts agents actually operate from). Only the second layer needs to be authoritative; the first is search convenience.

Open Knowledge Format (OKF) — Google Cloud, 2026-06-12

An open, vendor-neutral spec for exactly this: a directory of markdown files, one per concept, YAML frontmatter with a required type field plus optional title/description/tags, interlinked with normal markdown links. index.md and log.md are reserved filenames for directory listings and change history. It's a formalization of a pattern that was already emerging informally — worth knowing the name exists now, not worth over-indexing on adopting the spec literally for an internal repo of this size.
