Bottom line

→
The current setup — a cron-triggered process-inbox.sh that flocks a lockfile, runs a single Claude Code prompt over Mental Cache's inbox, and appends handled ids to a flat processed-ids file — is a sound minimal version of the pattern the automation-agent literature converges on (lock, checkpoint, idempotent re-run). It's missing two things every guide flags as necessary once a job runs unattended for months: a timeout/retry wrapper and a run summary that reaches you without you opening a log file. Beyond fixing those, the highest-value overnight work for this box splits cleanly: keep doing inbox triage + research reports for Mental Cache (already built), and extend the same report pipeline to a recurring Blockstream competitive/SEO watch. I could not audit Ledger or Ssaem — this VPS's working directories are scoped to /root/brain and /root/brain-reports only, so anything below is domain-informed, not repo-verified, for those two.

What's actually running today

The pipeline, as built

scripts/process-inbox.sh is invoked under flock -n /root/brain/.lock, so a second scheduled run exits immediately instead of racing the first — the right primitive, already in place. It sources .env and runs one claude -p call against prompts/process-inbox.md, logging to logs/process-<date>.log.

Idempotency is a flat file: memory/processed-ids.txt holds one handled entry id per line, matched by 8-character prefix. Simple and it works, but it's append-only and already several hundred lines with no compaction — fine for now, worth revisiting if this box ever tracks tens of thousands of captures.

Routing (from CLAUDE.md): a "research …" note gets a web-researched brief published as a formatted page via scripts/publish-report.sh (which rebuilds the index and pushes to the Vercel-deployed brain-reports repo); actionable phrasing becomes a shaped task; no-note captures get a light tidy pass and stay in the inbox.

Checked against best practice

The pattern converges across current guides on cron-based agents: lock against overlap, checkpoint progress so a restart resumes instead of repeating work, add retries so a transient API hiccup doesn't silently break the run, and end with a status notification rather than a log file nobody reads.

  • Have it: overlap lock, checkpointed idempotency, a routing spec that keeps runs small and defers deep research to explicit "research" captures.
  • Missing: no timeout on the claude -p call (a stuck run just holds the lock indefinitely — this session found exactly that: a stale lock and scratch files from an earlier interrupted run were still sitting in the repo), no retry wrapper around transient failures, and no "N processed / M failed, see report X" notification — success is currently only visible by opening the log or the DB.

What else this box could do overnight

Mental Cache / personal (verified scope)

This is the part already running and the part I can vouch for directly: inbox triage, research briefs for "research …" captures, and now formatted reports for anything substantial — the Fireblocks/Blockstream custody brief processed earlier today is the template. Natural next steps within this same repo:

  • A weekly rollup report (reports already worth reading as a set, not just individually) summarizing what got researched/tasked that week.
  • Resurfacing stale inbox entries — this run found a driving-test reminder that sat untouched since January; a periodic "these have been sitting N+ days" nudge would catch that pattern automatically.

Blockstream marketing (domain-informed, not repo-verified)

The report-publishing pipeline built for Mental Cache research generalizes directly to a recurring competitive/SEO watch, since it's the same shape of work (research → cited brief → published page):

  • Competitor moves in bitcoin custody/infra (the Fireblocks brief is a one-off version of this — could become a standing weekly or biweekly cadence).
  • Landing page / SEO technical checks (broken links, meta/schema drift, ranking movement on core enterprise-custody keywords) queued overnight, flagged only when something changed.
  • Ad creative sweeps (FB/Google ad libraries) for direct competitors, batched into the same report format rather than a raw dump.

I don't have this VPS wired to Blockstream's ad accounts, analytics, or a marketing repo, so none of this is currently actionable from here — it would need credentials/access scoped in the same way Mental Cache's Turso DB is, via a .env this box can read.

What it means for us

Keep the current architecture — it already matches the two things that matter most (overlap locking, idempotent re-runs). The cheapest next improvement is operational, not new capability: a timeout wrapper and a one-line Mental Cache digest entry per run, so a stuck or failed night surfaces itself instead of waiting to be noticed (as happened with this run's leftover lock and scratch files from an earlier interrupted session).

The highest-leverage new work, not just plumbing, is pointing the report pipeline that already exists at Blockstream competitive/SEO watch on a recurring cadence — it's the same research-then-publish shape as the Mental Cache reports, just aimed at a different inbox. That requires deciding what credentials this box should be trusted with beyond the Mental Cache DB.
