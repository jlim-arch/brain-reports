Buzz, and 5 agents worth launching on it for Blockstream

Jack Dorsey's Block shipped Buzz on 2026-07-21 — an open-source, Nostr-based Slack+GitHub replacement where AI agents get their own cryptographic identity and sit natively alongside humans in chat and code review. It's also the natural runtime for the "company brain for marketing agents" idea you and Michael Chen sketched in Slack (see the 2026-07-16 report) — Buzz is where those agents would actually live and act, not just where the fact sheets get stored.

Bottom line

→
Buzz is a workspace where every human and every AI agent holds its own cryptographic keypair, not an account owned by the platform — a human owner signs a narrowly-scoped authorization, and the agent then signs its own messages, code reviews, and workflow runs under that identity. Every action (chat message, review approval, git event) becomes one signed entry in a single hash-chained audit log. That's a genuinely good fit for Blockstream specifically — self-sovereign identity and a signed audit trail is the same ethos as the product line (Liquid, Green, Jade), so adopting it isn't just functionally useful, it's on-brand. It's also model-agnostic (works with Claude Code, Codex, Block's own goose), so it doesn't lock the team into one vendor. Caveat: it's four days old and explicitly early-stage — mobile and some workflow-approval features aren't finished — so this is a pilot-one-agent situation, not a replace-Slack-this-quarter situation.

What Buzz actually is

Identity & audit model

Every participant — human or agent — has a keypair that belongs to them, portable across any Nostr-compatible system, not tied to Buzz's servers. An agent's owner signs a scoped authorization (what it's allowed to do); the agent then signs its own work, so there's always a clear record of who authorized what.

Every message, reaction, workflow step, review approval, and git event is one signed event in a single hash-chained log — the same audit trail applies whether the actor was a person or a process.

What it replaces

Channels, threads, DMs, voice, media sharing — the Slack half. Git hosting and code review — the GitHub half. Automated workflows that agents can trigger and humans can approve — the missing glue between the two that today lives in a patchwork of Zapier/bots/CI.

Self-hostable, or Block runs a managed version. Desktop clients ship now for macOS/Windows/Linux; mobile and some workflow-approval UI are still in progress.

What it means for us

Don't launch all five at once — Buzz is four days old and self-admittedly not ready for a full team migration yet. The company-brain thread with Michael Chen already gives you the fact-sheet/guideline layer those agents would read from; Buzz would be where they actually post, review, and act with an identity people can hold accountable.

Suggested order

  • Pilot #2 (Launch QA) or #1 (Ship → Story) first — both have a clear, narrow scope and a visible pass/fail the first time they run, so you'll know quickly if the agent identity/permission model is trustworthy enough to build on.
  • Give each agent its own scoped keypair/authorization rather than one shared "marketing bot" identity — that's the whole point of Buzz's model, and it keeps the audit trail meaningful per-agent.
  • Decide self-host vs. Block's managed version early — self-hosting matches Blockstream's own self-custody positioning, but is more setup for a four-day-old platform.

Open questions

  • Does Blockstream engineering already have a Buzz opinion, or would marketing be the first team to touch it? Worth checking before building against it.
  • Is there a workflow-approval feature gap (mobile, some approval UI still unfinished per the coverage) that would block Launch QA specifically from having real teeth on day one?
